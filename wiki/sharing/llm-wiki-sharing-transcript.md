# LLM Wiki 搭建和使用心得

## 一、这东西是怎么来的

今年 4 月，Andrej Karpathy 在 GitHub 上发了一份 gist，几十行 Markdown，不到 2000 字。一周之内拿了 15000 多个 star，他的推文浏览量更是超过 1700 万。

Karpathy 是 OpenAI 的联合创始人之一，后来去了特斯拉做 AI 总监，目前在 Anthropic。他在 AI 圈子的影响力大，甚至有人称他是 AI **教导主任**。

这份 gist 的标题叫《LLM Wiki》。它描述的是一个**工作流模式**——用 LLM 持续维护一个知识库。

### 它要解决什么问题

先讲一个大家可能都熟悉的场景：RAG ，RAG 的工作流程可以分为三个简单的步骤：**检索（Retrieval）** 、**增强（Augmentation）**  、**生成（Generation）**。

假设我们公司有一个RAG系统，你问系统：“公司最新的报销流程是什么？”

1\. 【检索 (Retrieval)】 -> 去公司内部网盘、私有数据库里搜索“报销流程”相关的最新文档。 

2\. 【增强 (Augmentation)】 -> 把搜出来的文档内容（比如一页PDF说明）和你的问题打包在一起。 

3\. 【生成 (Generation)】 -> 把这个“打包好的资料包”丢给大模型。大模型看着资料，总结出准确的答案。

就像我们的小程序现在的小舟舟，应该就是这种模式。

这里面有一个问题：每次提问都是从零开始推导，没有积累。你问一个需要综合五份文档的问题，LLM 每次都要重新找到那些相关资料，重新拼凑。

RAG 就像一个**解释器**：每次提问都去临时检索原始文档，然后拼凑，用完就丢弃了。

Karpathy 提出的思路不一样。你把原始资料放在一个叫 `raw/` 的目录里，在你提问之前，先让LLM 做一次性的处理——读完、提取关键信息、更新相关的 wiki（**维基** 是一种允许**多人共同协作、自由创建和修改内容**的系统\*\*）\*\* 页面、建立交叉引用、标记矛盾、追加日志。这些处理好的页面放在 `wiki/` 目录里。之后你再提问，LLM 直接读这些已经整理好的页面来回答。

这就像一个**编译器**：资料进来的时候编译一次，之后基于编译产物推理。知识不是每次重新推导的，它是持续积累的。

### 为什么这件事以前没人做

维护知识库最烦人的就是整理——分类、打标签、更新交叉引用、保持不要过时、发现新资料和旧资料矛盾的地方等等。

人会放弃，因为维护成本增长得比知识价值快。你可能收藏了 200 篇文章，190 篇就那么躺着。

但LLM 不觉得烦，不会忘记更新某条交叉引用，一次能改 n 个文件。维护成本很低。**维护知识库这件事的等式变了。**

## 二、什么是 LLM Wiki

<br />

### 三层结构

整个系统就是三个东西：

**第一层：`raw/`** **— 原始资料**

你收集的文章、PDF、网页剪藏、截图，放在这个目录里。LLM 只读不写，这些是你的"事实来源"。wiki 如果出了问题，可以从 raw 重建。

**第二层：`wiki/`** **— 结构化知识**

全是 Markdown 文件。摘要页、概念页、对比分析、索引、日志。这一层完全由 LLM 创建和维护。你只读，LLM 写。

**第三层：Schema — 规则文件**

就是一份 `CLAUDE.md`（用 Claude Code 的话）或者 `AGENTS.md`（用 Codex 的话）。这份文件告诉 LLM：

- 这个 wiki 关于什么
- 目录怎么组织的
- 页面命名有什么规矩
- 处理新资料的时候要做什么、不能做什么

没有这份文件，LLM 就是一个通用聊天机器人。有了这份文件，LLM 变成了一个有纪律的 wiki 维护者。

这份规则文件不是一次写对的。你和 LLM 一起，用着用着慢慢改。

<br />

一句话总结：**人和 LLM 各自发挥长处，共同管理一个持续生长的知识库。**

人负责收集资料、判断资料质量、决定关注什么方向、问问题。LLM 负责所有的记账工作——读资料、写摘要、更新索引、建立交叉引用、标记矛盾、追加日志。

### 三种日常操作

**Ingest（摄入）**：把一份新资料丢进 `raw/`，让 LLM 处理。LLM 会读资料、写摘要、更新索引、更新相关的概念页和实体页、标记矛盾、追加日志。一份资料可能触发 N 个页面的更新。

**Query（查询）**：你提一个问题，LLM 先读 `index.md` 找到相关页面，然后跨页面综合信息，给出带引用的回答。好的回答可以写回 wiki——你无意中发现的一个联系、一次对比分析，不应该丢在聊天记录里消失。

**Lint（检查）**：定期让 LLM 做健康检查——有没有矛盾的论述、有没有没人链接的孤儿页面、有没有被新资料推翻的旧结论、有没有重要的概念还没建页面。这些维护工作人一定会跳过，LLM 不会。

### 一个帮助理解的类比

实际操作的时候，我一边开着 LLM agent，一边开着 Obsidian。LLM 根据对话编辑 wiki，我实时浏览结果——点链接、看图视图、读更新后的页面。

- Obsidian 是 IDE
- LLM 是程序员
- Wiki 是代码库
- 你是产品经理——决定做什么、验收质量

## 三、怎么搭一个 LLM Wiki

这一节是一个简洁的搭建指南。不追求详尽，追求够用。

### 前置条件

个人的意见：在动手之前，先问自己两个问题：

1. 你有一个**明确的问题**需要用知识库解决吗？
2. 你有**大量优质的相关资料**吗？

两个条件缺一个，建议先别搭。这是我的教训——后面心得那节会细说。

### 第一步：建目录

在你的电脑上建一个 git 仓库，里面放三个东西：

```
my-wiki/
  raw/          ← 原始资料，只读
    inbox/      ← 待处理的新资料丢这里
    sources/    ← 处理完的资料归档
  wiki/         ← LLM 维护的结构化知识
    index.md    ← 内容索引，LLM 每次查询先读这个
    log.md      ← 操作日志，按时间追加
  CLAUDE.md     ← 规则文件
```

### 第二步：选工具

你需要一个能读文件、能写文件、能遵循规则文件的 LLM agent。目前几个主流选择：

- **Claude Code**：终端里的 Claude agent，会自动读取项目里的 `CLAUDE.md`。兼容性最好。
- **Codex CLI**：OpenAI 的终端 agent，读取 `AGENTS.md`。
- **Cursor**：IDE 里带 agent 模式，也支持规则文件。

我自己用 Claude Code。选哪个取决于你习惯用什么。

### 第三步：写 Schema

这是最关键的一步。`CLAUDE.md` 写多好，wiki 就能跑多好。

最少要定义这几件事：

- 这个 wiki 解决什么问题
- 目录结构怎么组织
- Ingest 的时候要做什么（读 → 写摘要 → 更新索引 → 更新相关页 → 追加日志）
- Query 的时候要做什么（先读 index → 找到相关页 → 综合回答 → 好的回答写回 wiki）

Schema 是长出来的，不是一次写对的。先写一版能用的，跑几次 ingest，看 LLM 哪里做得不对，再改规则。

<br />

当然也可以把 Karpathy 的文档丢给 Agent，全部让 Agent 来给你干。

### 第四步：跑第一次 Ingest

1. 把第一份资料放进 `raw/inbox/`
2. 告诉 LLM："处理 raw/inbox/ 里的新资料"
3. 观察 LLM 做了什么——写了哪些页面、质量怎么样、有没有瞎编
4. 根据观察调整 `CLAUDE.md`
5. 放第二份资料，重复

### 日常节奏

- 有新资料 → 丢进 `raw/`，跑 ingest
- 有疑问 → 直接问 LLM，它会在 wiki 里搜索并综合回答
- 用了一段时间 → 跑一次 lint，做健康检查

### 你可能需要的工具

- **Obsidian**：浏览 wiki 的最佳工具。图视图能直观看到页面之间的关联，Dataview 插件能跑查询。wiki 就是用 Obsidian 风格的 `[[双向链接]]` 写的。
- Obsidian Web Clipper 将网页内容变成markdown 一键加入知识库。
- Obsidian 官方 skills。教会您的 Agent 使用 Obsidian CLI 和开放格式，包括 Markdown、Bases 和 JSON Canvas。
- **git**：wiki 就是一堆 markdown 文件，天然享受版本管理、分支、回滚。

## 四、我的实际使用体会

我用 LLM Wiki 搭了一个英语学习系统。这件事从 2026 年 5 月底开始，到现在一个月左右。下面是我真实的体会——好的、坏的都说。

### 一开始走了一段弯路

跟风 Karpathy 的 llm-wiki，但我想不出来我要用它干嘛。我试的第一个方向是一个大而全的"第二大脑"——把生活中的各种信息都往里丢，觉得以后会有用。

很快就放弃了。原因很简单：我没有那么强的求知欲去支撑这样一个泛化的知识库，我也没有收集大量优质资料的习惯。这件事变成了为建而建。

后来我换了一个方向：学英语。我的目标是 CEFR B2，大概是一年时间，每天投入 30 分钟左右。

这个场景恰好满足那两个前置条件：

- **明确的问题**：我现在大概认识 1000 个基础词，要系统性地提升到 B2 水平
- **大量优质资料**：Oxford 3000/5000 词表、BBC Learning English 的听力和 transcript、British Council 的分级阅读、Cambridge 的语法资料——这些全部是 A 级和 B 级来源，免费且高质量

方向对了之后，wiki 才开始真正产生价值。

### 最核心的价值：英语水平画像

这个系统帮我做了一件传统学习方法很难做到的事：持续追踪我自己的英语水平。

每天完成学习任务之后，LLM 批改我的输出——填空、造句、改错、自由写作。然后根据我的表现更新 wiki：

- **词汇掌握程度**：每个词有三个等级——L1 刚学、L2 能在控制练习中用对、L3 自由输出中也能用对。如果哪天自由输出拼错了，等级会回退。这不是"学过了就算"，是真实的掌握追踪。
- **语法错误模式**：同一个错误出现了几次、是在控制练习中还是自由输出中出现的、连续几次正确之后可以降为抽查。比如 `want to do` 这个结构，我第 5 次才真正记住——前 4 次都在不同场景下漏了 `to`。系统记录了每一次，直到连续 3 次正确才标记为"已修复"。
- **弱项清单**：根据实际错误累积出来的弱项列表，每一条都有具体的出现日期和修复进展。不是"我觉得我语法不好"这种模糊的自我感觉，而是"现在完成时 + 具体时间混用，06-10 修复，06-18 回退，06-22 再次修复"这种精确的追踪。

### LLM 根据画像定制学习内容

这套画像不只是记录，它直接驱动了每天的学习计划。

举个例子：某天我的自由输出中出现了语法错误，系统会这个错误类型（内部代号"Q7 型"）。第二天的计划里，就会出现 1-2 道专门针对这个语法的改错题，同时新内容的分量会被压轻，因为今天有结构性错误需要先修。

另一天，如果我的错误很少（0-1 个结构性错误），系统不会加更多复习，而是触发"前摄语法"——主动引入一个新的语法点，比如 present perfect vs past simple 对比。这是利用我当天的好状态来推进新知识。

这种根据实际表现动态调整计划的能力，是任何静态教材和通用学习 App 做不到的。

### 诚实地说说风险

LLM 会产生幻觉——它可能编造不存在的语法规则，可能错误地标记一个词的 CEFR 等级，可能在解释某个用法的时候把因果关系搞反。

在学习场景下，这个问题尤其致命。如果我不是英语母语者，我没办法一眼识别 LLM 教我的东西是不是对的。错的东西被写进 wiki、反复强化，后果比什么都不学更糟糕。

我采取了一些防范措施：

- 所有原始资料来自权威来源（Oxford、BBC、British Council、Cambridge），wiki 里的知识必须能追溯到某个 raw/ 里的源文件
- CLAUDE.md 明确写了来源分级（A 级权威 / B 级优质 / C 级真实材料 / D 级 LLM 生成），LLM 不能把 D 级内容当权威
- 定期跑 lint 检查矛盾

但说实话：这些措施能降低风险，不能消除风险。我本身只是一个英语学习者，不是英语老师，有些错误我可能到现在都不知道。这是使用 LLM Wiki 需要保持的一个基本警惕。

### LLM Wiki 不是银弹

它只是一种知识管理和应用方式。它适合的情况很明确：你有一个具体的目标，有大量可以信赖的资料。

别为了用而用。先有真实的需求，再考虑 LLM Wiki 能不能帮上忙。

## 五、总结

LLM Wiki 这件事的本质很简单：人和 LLM 各展所长，共同维护一个会生长的知识库。

人做的事没变——收集、判断、思考、提问。变的是 LLM 把人类一直做不好也不想做的"记账"工作接了过去。交叉引用不会忘了更新，摘要不会过时，矛盾不会被默默忽略。

它不替代你的思考。系统负责记账，思考仍然是你的。

如果你有一个明确的问题，手头有足够多值得信赖的资料，这件事值得试一下。

# LLM Wiki 搭建和使用心得 — 分享大纲

## 一、产生背景（约 3 分钟）

### 1.1 源头：Karpathy 的一份 75 行 gist
- 2026 年 4 月，Andrej Karpathy 在 GitHub 上发了一份不到 2000 字的 gist
- 一周内 15,000+ stars，X 浏览量超 1700 万
- 它不是一个工具、不是一个产品，是一个**工作流模式**

### 1.2 它要解决什么问题
- 传统的 RAG（检索增强生成）：每次提问都从原始文档里临时翻、临时拼
  - 类比：解释器模式 —— 每次运行重新求值
- LLM Wiki：新资料进来就编译好，之后直接基于编译产物推理
  - 类比：编译器模式 —— 一次编译，持续复用
- 核心洞察：人类放弃维护知识库，不是因为懒，是因为维护成本增长快于知识价值。LLM 改变了这个等式。

## 二、什么是 LLM Wiki（约 5 分钟）

### 2.1 一句话定义
- 人和 LLM 各自发挥长处，共同管理一个持续生长的知识库

### 2.2 三层架构（核心概念）
- **raw/** — 原始资料，只读不改。你负责收集，LLM 只能读取
- **wiki/** — 结构化知识。LLM 全权维护：写页面、做交叉引用、标记矛盾
- **Schema（CLAUDE.md）** — 规则文件。告诉 LLM"怎么维护这个 wiki"。你把通用聊天机器人变成有纪律的 wiki 维护者

### 2.3 三种操作
- **Ingest（摄入）**：新资料进来 → LLM 读取 → 更新 10-15 个相关页面
- **Query（查询）**：提问 → LLM 读 index.md 定位页面 → 综合回答并附引用。好的回答可回写进 wiki
- **Lint（检查）**：定期扫描矛盾、孤立页面、过时内容、缺失交叉引用

### 2.4 一个直观类比
- Obsidian = IDE
- LLM = 程序员
- Wiki = 代码库
- 你 = 产品经理（提需求、审质量）

## 三、落地指南：怎么搭一个 LLM Wiki（约 8 分钟）

### 3.1 前置条件（最重要的一步）
- 你有**一个明确的问题**需要用知识库解决
- 你有**大量优质的相关资料**
- 缺任何一个，先别搭

### 3.2 三步搭建
1. **建目录**：`raw/`（放资料）、`wiki/`（LLM 维护）、一份 `CLAUDE.md`（规则）
2. **选工具**：Claude Code / Codex CLI / Cursor Agent 模式。核心要求：能读文件、能写文件、能遵循规则文件
3. **写 Schema**：定义 wiki 结构、命名约定、工作流程。这是最关键的一步——schema 质量决定 wiki 上限

### 3.3 第一次 Ingest
- 放一份资料进 raw/ → 告诉 LLM "处理这份资料" → 观察 LLM 做了什么
- 根据结果调整 schema → 再试下一份
- Schema 是**长出来的**，不是一次写对的

### 3.4 日常操作节奏
- 有新资料 → ingest
- 有问题 → query（先读 index，再读相关页，综合回答）
- 定期 → lint（健康检查）

### 3.5 工具推荐（简要）
- 编辑器：Obsidian（图视图、双向链接、Dataview 插件）
- 搜索：wiki 小时靠 index.md，大了上 qmd（本地 BM25/向量混合搜索）
- 版本管理：git（wiki 就是一堆 markdown 文件）

## 四、个人心得（约 8 分钟）

### 4.1 走弯路：大而全的"第二大脑"失败了
- 一开始想做全个人知识库，什么都往里丢
- 失败原因：没有明确问题、没有大量优质资料
- 学英语这个场景恰好两个条件都满足：B2 目标是明确的，Oxford/BBC/British Council 资料是海量且优质的

### 4.2 最核心的价值：英语水平画像
- 系统根据每天的错误累计，逐渐建立了我个人的英语能力档案
  - 哪些语法规则掌握了、哪些反复出错
  - 每个词汇的掌握程度（刚学 / 能用 / 已内化）
  - 错误模式追踪（同一个错误出现了几次、是在控制练习还是自由输出中出现的）
- LLM 根据这个画像定制每天的学习内容——不是通用的学习计划，是针对我今天弱项的 30 分钟练习

### 4.3 诚实面对风险
- LLM 可能生成错误的、过时的、幻觉的知识
- 在学习场景下这尤其致命——你不知道你学到的是不是对的
- 防范手段：原始资料必须来自权威来源（A-level sources），wiki 内容可追溯到 raw/，定期 lint 检查矛盾
- 但说实话：我也不能 100% 保证 wiki 里没有错误，因为我本身就是学习者

### 4.4 LLM Wiki 不是银弹
- 它只是一种知识管理和应用方式
- 适用条件：明确的问题 + 大量优质资料
- 不适合：模糊的需求、资料匮乏的领域、一次性查询

---

## 五、总结（约 2 分钟）

- LLM Wiki 的本质：人和 LLM 各展所长，共同维护一个会生长的知识库
- 它不替代你的思考，它替代的是你不想做的记账工作
- 如果你有一个明确的问题和足够的优质资料，值得一试
